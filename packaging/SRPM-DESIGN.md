# servicectl Stack SRPM Design

## 1. Goal and Scope

The project will ship one self-contained source RPM that embeds the source
archives needed to build the complete servicectl runtime stack on Fedora 44
and newer.

The source RPM produces multiple binary RPMs rather than one monolithic RPM.
This keeps file ownership, licenses, library dependencies, upgrades, and
debug packages understandable while preserving a one-command installation
path through a meta package.

The first release has these boundaries:

- Fedora 44 and newer.
- Native RPM builds for the current architecture.
- All third-party source archives and Go module sources are embedded in the
  SRPM, so rebuilding the SRPM does not require network access.
- Installation does not replace PID 1 and does not automatically start dinit
  or an s6 supervision tree.
- systemd remains available as the source of unit definitions and for host
  utilities such as `busctl` and `journalctl`.
- Runtime-generated Dinit, s6-rc, D-Bus activation, socket, state, and log
  files remain owned by servicectl at runtime rather than being pre-generated
  by RPM scriptlets.

## 2. Locked Source Set

The source lock file records the URL, version, SHA-256 digest, and archive
filename for every embedded source. License metadata is documented here and
copied from each verified source archive into the relevant binary package.

The preparation script must reject dirty source archives unless explicitly
allowed, and must verify every third-party digest before creating the SRPM.

## 3. Build Dependency Order

The single spec builds components in this order:

1. skalibs
2. execline, using the staged skalibs headers, libraries, and sysdeps
3. s6, using staged skalibs and execline
4. s6-rc, using staged skalibs, execline, and s6
5. dinit, independently of the Skarnet stack
6. servicectl and its production helper binaries, using the vendored Go tree

All components install into one private build root during `%install`. Later
subpackages divide that build root by file ownership.

Skarnet packages do not support out-of-tree builds. `%prep` therefore creates
one writable source directory per component, and each component is configured,
built, and installed in place.

## 4. Binary RPM Layout

The SRPM name is `servicectl-stack`. It produces these binary packages.

### `servicectl-stack`

Meta package only. It owns no runtime files and requires exact versions of:

- `servicectl`
- `servicectl-dinit`
- `servicectl-s6`

After the RPMs are published in a repository, installing this package is the
supported one-command deployment path:

```bash
sudo dnf install servicectl-stack
```

For freshly built local RPMs, all runtime packages are passed to one DNF
transaction as documented in `packaging/README.md`; DNF does not automatically
scan sibling RPM files when only the local meta-package path is supplied.

### `servicectl`

Owns the project runtime:

- `/usr/bin/servicectl`
- `/usr/bin/sys-notifyd`
- `/usr/bin/sys-cgroupd`
- `/usr/bin/sys-logd`
- `/usr/bin/sys-propertyd`
- `/usr/bin/sysvisiond`
- `/usr/bin/sys-orchestrd`
- `/usr/lib/servicectl/socket-holders.d/dbus.conf`
- `/usr/lib/tmpfiles.d/servicectl.conf`
- `/var/lib/servicectl`
- `/usr/share/licenses/servicectl/LICENSE`

The package does not include the test-only binaries `notify-echod`,
`notify-sleeper`, or `test-envd`.

Runtime requirements include the shell and host tools called by production
code:

- `bash` for `/bin/sh`
- `coreutils` for `chown`
- `procps-ng` for `ps` and `pgrep`
- `systemd` for `busctl` and `journalctl`
- `dbus-common` for the system D-Bus activation directory

The package requires both `servicectl-dinit` and `servicectl-s6`: runtime
control invokes `dinit`/`dinitctl`, while persistent enablement and group
orchestration invoke the s6 and s6-rc command sets.

### `servicectl-dinit`

Owns Dinit executables, manuals, and the persistent service definition root:

- `/usr/bin/dinit`
- `/usr/bin/dinitctl`
- `/usr/bin/dinit-check`
- `/usr/bin/dinit-monitor`
- Dinit manual pages
- `/etc/dinit.d`

Dinit is built as a service manager, not as PID 1. Shutdown aliases are not
installed, preventing conflicts with Fedora's systemd-owned `shutdown`,
`halt`, `poweroff`, and `reboot` commands.

The package requires `libcap` when capability support is enabled. Fedora build
flags and linker hardening flags are passed through Dinit's configure/build
variables.

### `servicectl-s6`

Owns the executables required by the servicectl s6 backend and its persistent
source database root:

- execline command set, including `/usr/bin/execlineb`
- s6 command set, including `/usr/bin/s6-svscan`
- s6-rc command set, including `/usr/bin/s6-rc`, `/usr/bin/s6-rc-compile`, and
  `/usr/bin/s6-rc-update`
- `/s6/rc`

This package requires the exact matching runtime library packages generated by
the same SRPM. It does not create or start `/run/s6/state`; the operator must
start an s6 supervision tree explicitly before enabling live updates.

### Runtime and development library packages

The stack exposes conventional shared libraries and separates development
files:

- `servicectl-skalibs-libs`
- `servicectl-execline-libs`
- `servicectl-s6-libs`
- `servicectl-s6-rc-libs`
- `servicectl-stack-devel`

The `-libs` packages own versioned shared objects and their SONAME links. The
`-devel` package owns headers, static archives, unversioned linker symlinks,
pkg-config files, and skalibs sysdeps.

The packages use namespaced RPM names but provide their conventional
capabilities where safe. They conflict with other packages that own the same
unprefixed `/usr/bin` commands or shared-library paths. This is necessary
because Fedora 44 does not currently provide these command files, while local
or COPR installations may already own them.

## 5. Filesystem Ownership

### Persistent paths owned by RPM

```text
/etc/dinit.d
/s6/rc
/var/lib/servicectl
/usr/lib/servicectl/socket-holders.d
```

`/var/lib/servicectl/properties/state` is created by `sys-propertyd`; RPM owns
only the parent directory.

### Volatile paths created by tmpfiles.d

`servicectl.conf` creates only stable parent directories:

```text
d /run/servicectl 0755 root root -
d /run/servicectl/managed 0755 root root -
d /run/servicectl/sysvision 0755 root root -
d /run/servicectl/sys-cgroupd 0700 root root -
d /run/servicectl/logspill 0755 root root -
d /run/dinit.d 0755 root root -
d /run/dinit.d/generated 0755 root root -
d /run/s6 0755 root root -
```

The package does not pre-create `/run/s6/state` because that directory denotes
a live s6-rc instance and must be initialized by the operator's supervision
tree.

Per-service state directories, notify sockets, D-Bus trigger sockets, API
sockets, generated Dinit services, and compiled s6 databases are created and
removed by servicectl at runtime.

The `sys-cgroupd` tmpfiles entry creates only a root-owned runtime directory.
It does not create `/sys/fs/cgroup/servicectl.slice`, migrate a process, enable
a cgroup controller, or start the daemon. The packaged Dinit definition is
available for explicit operator startup and has no RPM scriptlet activation.
When the explicitly started daemon finds no visible cgroup2 mount, its default
runtime policy may safely mount cgroup2 at `/sys/fs/cgroup` only when that path
is a real empty directory. It never remounts or unmounts cgroup2, mounts over
existing content, or enables controllers. Administrators can add
`--no-auto-mount` to the local Dinit command.

### Mutable D-Bus activation files

Earlier code wrote generated activation files below
`/usr/share/dbus-1/system-services`. RPM-managed `/usr` must not be used for
mutable runtime state. Packaging implementation therefore includes a small
application change:

- system-mode generated activation files move to
  `/etc/dbus-1/system-services`;
- the package owns that directory if it is absent;
- vendor activation files remain readable from
  `/usr/share/dbus-1/system-services` through the normal D-Bus search path.

This keeps generated files local-administrator state and prevents them from
being silently outside RPM ownership under `/usr`.

### User-mode paths

User-mode directories remain lazily created by servicectl:

- `~/.config/dinit.d`
- `$XDG_RUNTIME_DIR/dinit.d/generated`
- `/run/user/$UID/servicectl`
- `~/.local/share/dbus-1/services`
- `~/.config/servicectl`

No RPM scriptlet creates user home directories.

## 6. Build Configuration

### Skarnet packages

All Skarnet components install under Fedora's `/usr` hierarchy and use
shared libraries:

```text
--prefix=/usr
--bindir=/usr/bin
--libdir=/usr/lib64
--dynlibdir=/usr/lib64
--includedir=/usr/include
--enable-shared
--disable-allstatic
--enable-pkgconfig
```

skalibs additionally uses:

```text
--sysdepdir=/usr/lib64/skalibs/sysdeps
```

The exact `%{_libdir}` macro is used instead of hard-coding `lib64`, keeping
the spec architecture-correct. Build and install commands receive
`DESTDIR=%{buildroot}` and Fedora compiler/linker flags.

execline is not built as a multicall binary in the first version. Individual
commands make file-level ownership and diagnostics clearer, and compatibility
is more important than the small disk saving.

### Dinit

Dinit is configured with:

```text
--bindir=/usr/bin
--sbindir=/usr/bin
--mandir=/usr/share/man
--syscontrolsock=/run/dinitctl
```

The exact configure spelling is verified against the locked archive during
implementation. `BUILD_SHUTDOWN=no` is mandatory.

### servicectl

The source preparation script runs `go mod vendor` in a clean temporary copy.
The generated vendor tree is part of the servicectl source archive embedded in
the SRPM.

The spec builds with:

```text
GOFLAGS=-mod=vendor
CGO_ENABLED=0
go build -trimpath -buildvcs=false -ldflags "-s -w ..."
```

Production binaries are built explicitly rather than using `go build ./...`,
which would also build test helpers.

## 7. Source RPM Structure

The repository gains this layout:

```text
packaging/
  SRPM-DESIGN.md
  sources.lock
  servicectl-stack.spec
  servicectl.tmpfiles
  build-srpm.sh
  fetch-sources.sh
  README.md
```

The old `packaging/skalibs.spec` is removed after its relevant build flags are
merged into `servicectl-stack.spec`.

`sources.lock` is a simple machine-readable TSV file:

```text
component<TAB>version<TAB>sha256<TAB>url<TAB>filename
```

`fetch-sources.sh` downloads only missing archives, verifies every digest, and
places them in a caller-selected source cache. It never trusts a file solely
because the filename already exists.

`build-srpm.sh` performs the one-command preparation flow:

1. validate required tools;
2. export a clean servicectl source snapshot;
3. vendor Go modules into that snapshot;
4. download and verify locked third-party archives;
5. copy all sources, spec, and auxiliary files into a temporary RPM topdir;
6. run `rpmbuild -bs`;
7. print the resulting SRPM path.

Example:

```bash
./packaging/build-srpm.sh
```

Optional arguments select the output directory, source cache, and an explicit
servicectl version/release. Network access is needed only while populating an
empty cache; rebuilding from an existing cache is offline.

## 8. Version and Release Policy

The SRPM has one stack version derived from servicectl, while each bundled
component retains its upstream version in the corresponding binary package:

```text
servicectl-stack-0.1.0-1.fc44
servicectl-0.1.0-1.fc44
servicectl-dinit-0.22.1-1.fc44
servicectl-s6-2.15.0.0-1.fc44
servicectl-skalibs-libs-2.15.0.0-1.fc44
servicectl-execline-libs-2.9.9.1-1.fc44
servicectl-s6-libs-2.15.0.0-1.fc44
servicectl-s6-rc-libs-0.6.1.1-1.fc44
```

Because the repository currently has no release tags and no application
version command, the first packaging implementation introduces an explicit
`servicectl_version` macro in the spec. Snapshot builds may append a date and
short commit to `Release`; dirty trees are rejected by default.

## 9. RPM Scriptlets and Safety

Scriptlets are intentionally minimal:

- `%post` runs `systemd-tmpfiles --create servicectl.conf` when available.
- No scriptlet starts, stops, enables, or reloads dinit, s6, servicectl, D-Bus,
  `sys-cgroupd`, or systemd services.
- No scriptlet or tmpfiles rule mounts cgroup2; safe auto-mount is a daemon
  startup policy only.
- No scriptlet modifies PID 1, kernel command lines, initramfs, alternatives,
  or `/sbin/init`.
- Uninstall removes only RPM-owned files. Runtime state in `/var/lib` follows
  normal RPM semantics and is not recursively deleted by shell scriptlets.

The package may print a post-install note showing the explicit commands needed
to start a Dinit control process or initialize an s6 tree, but it does not run
those commands automatically.

## 10. Validation

### `%check`

Safe buildroot checks are:

- verify every embedded source digest;
- run upstream Skarnet build checks when provided and non-privileged;
- run `make check` for Dinit;
- run `go test -count=1 ./...` for servicectl;
- run the unprivileged `sys-cgroupd` integration-script self-test without
  touching host cgroupfs;
- assert all eight production Go binaries exist;
- run `dinit-check` against generated fixture services;
- compile a minimal s6-rc fixture with the just-built `s6-rc-compile`;
- inspect ELF dynamic dependencies and reject references to buildroot paths;
- verify no package file is installed under `/usr/local`, `/bin`, or `/lib`;
- verify the RPM payload contains no mutable `/run` files.

Host integration scripts under `scripts/test-*.sh` are not run in `%check`
because they start and stop real host services and write host runtime state.
They remain a separate post-build test stage in a disposable VM.

`scripts/test-cgroupd-integration.sh --self-test` is the exception: it validates
only script syntax, dependencies, and the explicit delegated-root contract. Its
real mode requires `--cgroup-root PATH`, root privileges, and a writable cgroup
v2 subtree; `%check` never invokes that mode.

### SRPM acceptance test

The packaging CI path is:

```bash
./packaging/build-srpm.sh
mock -r fedora-44-aarch64 --rebuild <result.src.rpm>
dnf install <all-runtime-rpm-paths-in-one-transaction>
servicectl version
dinit --version
s6-rc help
```

The installation test must also assert that PID 1 is unchanged and no new
long-running process was started by RPM scriptlets.

## 11. Known Implementation Work

The packaging implementation includes these repository changes:

1. add the top-level multi-package spec and source-lock tooling;
2. add the tmpfiles configuration;
3. vendor Go modules only in the generated source archive, not permanently in
   the development repository;
4. change system-mode generated D-Bus activation output from mutable `/usr` to
   `/etc/dbus-1/system-services`;
5. add a stable servicectl package version input suitable for RPM builds;
6. merge and retire `packaging/skalibs.spec`;
7. document build, rebuild, install, upgrade, uninstall, and offline-cache
   workflows in `packaging/README.md`.

This design deliberately does not add systemd units that auto-start dinit or
s6. A separate opt-in integration package can be designed later without
changing the source RPM's core dependency and filesystem layout.
